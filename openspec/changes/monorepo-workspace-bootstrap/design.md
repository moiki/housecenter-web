# SDD Design — Monorepo Workspace Bootstrap

## Change name
`monorepo-workspace-bootstrap`

## Status
`design` (2026-07-12)

---

## Technical approach (one line)

Convert `housecenter-web` in-place into a pnpm-workspaces + Turborepo monorepo by
`git mv`-ing the whole app tree into `apps/web/` as a unit (alias survives unedited) and
adding additive root orchestration config — a pure structural + config change with web
behavior unchanged by contract.

## Architecture decisions

### 1. In-place conversion of `housecenter-web` (not a fresh repo)
**Choice**: Restructure the existing repo on branch `feat/monorepo-workspace-bootstrap`; `.git`
and `openspec/` stay at root.
**Alternatives**: New `housecenter-frontend` repo + copy the app in.
**Rationale**: `git mv` preserves per-file history/blame across the move; a fresh repo would
sever it and orphan the existing openspec change trail. The move is fully reversible (revert
the branch), so in-place carries no lock-in.

### 2. `node-linker=hoisted` now (not pnpm's default `isolated`)
**Choice**: Set `node-linker=hoisted` in root `.npmrc` in this change.
**Alternatives**: Keep the default isolated/symlinked `.pnpm` store; defer to the mobile change.
**Rationale**: Metro (future Expo `apps/mobile`) historically fights pnpm's symlinked store,
requiring bespoke `watchFolders`/`nodeModulesPaths`/`disableHierarchicalLookup`. A flat tree
pre-empts that — this is the master plan's #1 infra risk, retired proactively. Cost to Vite/web
is ~zero: Vite is indifferent to `node_modules` layout and a flat tree is closer to today's
single-root shape. Cheapest to prove now while the graph is tiny.

### 3. Turborepo `tasks` pipeline shape
**Choice**: `tasks: { build: { dependsOn:["^build"], outputs:["dist/**"] }, lint: {}, dev: { cache:false, persistent:true } }`.
**Alternatives**: deprecated `pipeline` key (removed at Turbo 2.0); add `lint dependsOn ^build`; add a `test` task.
**Rationale**: `tasks` is correct for Turbo 2.10.4. `build ^build` topo-orders future
cross-package builds; `dist/**` is the only cached artifact. `lint` has **no** `dependsOn`
because ESLint here extends `tseslint.configs.recommended` (not `recommendedTypeChecked`) — it
is not type-aware, so it needs no upstream build. **No `test` task** yet: no package has a test
script and `strict_tdd: false`; adding one would make `pnpm -w test` fail on an empty pipeline.

### 4. `tsconfig.base.json` extraction — hoist vs leaf
**Choice**: Hoist genuinely-shared flags to `tsconfig.base.json` and **remove them from the
leaves**, which now carry only leaf-specific keys + `"extends": "../../tsconfig.base.json"`.
Hoisted: `target, module, moduleResolution, allowImportingTsExtensions, verbatimModuleSyntax,
moduleDetection, skipLibCheck, strict, noUnusedLocals, noUnusedParameters, erasableSyntaxOnly,
noFallthroughCasesInSwitch, ignoreDeprecations`. Left in leaves: `lib, types, jsx, noEmit,
baseUrl, paths, tsBuildInfoFile, include`.
**Alternatives**: (a) proposal shorthand "+1 line each" — add `extends` but leave every option
duplicated in the leaf; (b) no base file at all.
**Rationale**: A base that duplicates every leaf option is dead config; real DRY requires the
leaves to shed the hoisted keys so base is the single source of truth for core/mobile later.
`noEmit` is deliberately **left in the leaves** — the future `packages/core` will emit `.d.ts`,
so `noEmit` must not live in a base it extends. **`baseUrl:"."` + `paths:{"@/*":["src/*"]}`
stay in `tsconfig.app.json`**: `baseUrl` is relative to the config file's own directory, so after
the move `@/*` resolves to `apps/web/src/*` with zero edits — matching Vite's `__dirname`-relative
alias. Both alias definitions travel intact precisely because they were never root-relative.
*(This realizes the proposal's "+1 line" shorthand as hoist-and-trim; the leaf diff is a small
`compilerOptions` rewrite, not a single line — see Config file contents. sdd-tasks should size
accordingly.)*

### 5. Minimal placeholders for `core` and `mobile`
**Choice**: One `package.json` each (`{"name":…,"private":true,"version":"0.0.0"}`), no `src/`,
tsconfig, or build script.
**Alternatives**: Empty dirs (invisible to the workspace glob); fully scaffolded packages.
**Rationale**: A lone `package.json` is the smallest unit the `packages/*`/`apps/*` globs need to
materialize a valid four-importer lockfile and prove the graph resolves. Real content is out of
scope (core-extraction, mobile-app-foundation). No `build` script means `turbo run build` cleanly
skips them — proving pipeline wiring without inventing work.

### 6. Keep pnpm 9.15.9 (EOL) pinned
**Choice**: Move the exact `packageManager: "pnpm@9.15.9+sha512…"` string to the **root**
`package.json`; drop it from `apps/web/package.json`.
**Alternatives**: Bump to pnpm 11 in this change.
**Rationale**: A package-manager major bump is orthogonal and separately risky (Node ≥22 floor,
lockfile format shifts) — folding it in would blur "did the restructure break the build?" It gets
its own fast-follow **pnpm-upgrade** change. The root owns the pinned version so every workspace
package inherits one toolchain.

### 7. `.claude/skills` Option A relocation + new root `CLAUDE.md`
**Choice**: `git mv` the 4 web skills into `apps/web/.claude/skills/` unedited; add a small
**new** root `CLAUDE.md` describing the monorepo shape and pointing to `apps/web/CLAUDE.md`.
**Alternatives**: Rewrite each skill's `src/…` paths to `apps/web/src/…`, or duplicate skills to root.
**Rationale**: The skills cite app-root-relative paths (`src/pages/…`); moving them beside the code
keeps every path correct when cwd = `apps/web`, with **zero content edits** (honors "no behavior
change"). Rewriting 6 prose files is scope creep and error-prone. Residual risk: invoking a skill
from the **monorepo root** cwd could resolve paths against the wrong dir — a non-blocking manual
follow-up, cheaply fixed later if it bites.

### 8. Node `engines` floor
**Choice**: Add root `engines.node = "^20.19.0 || ^22.12.0 || >=24.0.0"`.
**Alternatives**: No engines field (status quo).
**Rationale**: Vite 8 requires Node 20.19+/22.12+; declaring it documents the floor and warns on
mismatched environments before Expo layers its own Node requirement. New but low-risk (advisory,
not enforced unless `engine-strict`).

## Migration sequence (ordered apply steps)

1. **Verify clean tree** (apply phase has Bash; exploration did not). `git status` → clean or
   only expected changes; `git branch --show-current` → `feat/monorepo-workspace-bootstrap`;
   `git log --oneline -1`. **Abort** on anything unexpected — do not `git mv` a dirty tree.
2. **Create root files**: `package.json`, `pnpm-workspace.yaml`, `.npmrc`, `turbo.json`,
   `tsconfig.base.json`, root `CLAUDE.md`; `mkdir -p apps/mobile packages/core` then write the two
   placeholder `package.json`s.
3. **`git mv` the app tree** into `apps/web/`: `mkdir -p apps/web`, then `git mv` each tracked
   path — `src public index.html vite.config.ts eslint.config.js tsconfig.json tsconfig.app.json
   tsconfig.node.json package.json .env.example README.md CLAUDE.md docs` → `apps/web/`. Plain
   `mv .env apps/web/.env` (untracked). Do **not** move `node_modules dist pnpm-lock.yaml openspec
   .gitignore .git`.
4. **Apply the 3 edits** in `apps/web/`: `package.json` (drop `packageManager`, `name`→`web`);
   `tsconfig.app.json` and `tsconfig.node.json` (add `extends`, remove hoisted keys per above).
5. **`git mv` `.claude/skills`** (Option A): `git mv .claude apps/web/.claude` — 4 skills relocate
   as-is, no content edits.
6. **`pnpm install`** from root → regenerates a single multi-importer `pnpm-lock.yaml` and a hoisted
   root `node_modules`.
7. **Build / lint / verify** (next section).

## Verification

| Command | Expected |
|---|---|
| `pnpm install` (root) | One `pnpm-lock.yaml`, importers `{ ., apps/web, packages/core, apps/mobile }`; root importer has only `turbo` |
| `pnpm --filter web build` | `tsc -b && vite build` green; `apps/web/dist/` produced — proves `@/*` alias + build survived the move |
| `pnpm --filter web lint` | Zero new findings (pure move) |
| `pnpm -w build` (`turbo run build`) | Web builds; core/mobile skipped (no `build` script); `.turbo` cache created |
| `git diff --stat` / `git status` | Only renames (`R`) + the 3 deliberate edits + new root files; no `src` content churn |
| Manual (non-blocking) | `pnpm --filter web dev` smoke; `.claude/skills/*` invocation from both root and `apps/web` cwd |

## Rollback

Fully reversible — structural rename + additive config, no data/state migration:
- Revert the `feat/monorepo-workspace-bootstrap` branch (or its merge commit): `git mv` unwinds to
  the flat layout with history intact.
- Delete `node_modules/` + `pnpm-lock.yaml` and `pnpm install` against the reverted
  `package.json` → the original single-importer lockfile regenerates.
- No source, CI, or deploy config was edited, so nothing downstream needs unwinding.

## Risks / open items for sdd-tasks

- **tsconfig leaf diffs are a small `compilerOptions` rewrite**, not the proposal's literal
  "+1 line each" — hoist-and-trim is the correct realization (§4). Still tiny, well under the
  400-line human-review budget, but size the tasks against the shown JSON.
- **`.turbo/` cache** is generated at `apps/web/.turbo` (and root). The existing `.gitignore` does
  not list it. Recommend adding `.turbo` to root `.gitignore` (one line) — otherwise it shows up
  untracked after the first `pnpm -w build`. Flag for a task; keep it in-scope-adjacent.
- **Root-cwd skill discovery** (§7) is unverified without Bash — a manual, non-blocking follow-up,
  not a build gate.
- **`.env` is untracked** → must be a plain `mv`, not `git mv`; easy to miss in an automated apply.
- **Working-tree cleanliness** is a hard precondition (step 1) — the apply phase must gate on it.
